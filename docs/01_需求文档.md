# 需求文档

**产品名称：** DietSearch · 膳食导向餐厅检索系统
**版本：** v1.0
**更新日期：** 2026-05-22

---

## 1. 概述

### 1.1 问题背景

随着健康意识提升与多元饮食文化的普及，用户在外出就餐时面临越来越多的饮食限制与偏好需求：素食主义、无麸质、低碳水、清真、生酮、过敏原回避等。现有餐厅搜索工具（如大众点评、Google Maps）存在以下核心痛点：

- **膳食筛选维度单一**：仅支持菜系、价格等粗粒度标签，无法精准匹配具体饮食方案；
- **信息准确性不足**：菜单中的配料、烹饪方式对有过敏原或医疗饮食需求的用户至关重要，但现有平台几乎不收录；
- **检索相关性差**：用户以自然语言描述饮食需求（如"高蛋白低脂肪适合增肌"），现有系统无法理解语义；
- **个性化匹配缺失**：不同用户的饮食限制差异极大，缺乏基于个人档案的个性化排序。

### 1.2 产品愿景

构建一套以**信息检索（IR）技术**为核心的膳食导向餐厅搜索系统：用户以自然语言描述饮食需求，系统通过文档索引、语义检索、多维排序等 IR 方法，精准返回符合饮食方案的餐厅与菜品，并支持结果解释。

### 1.3 当前范围

本文档覆盖 v1.0 核心功能：
- 餐厅与菜品数据的离线索引构建
- 基于饮食关键词的全文检索（BM25 / TF-IDF）
- 饮食分类 Faceted 搜索（素食、清真、无麸质等）
- 自然语言查询解析与查询扩展
- 多维度相关性排序
- 搜索结果展示与解释
- 用户饮食档案与个性化重排序

**不在本期范围：**
- 餐厅预订与支付
- 用户评论生成
- 商家入驻管理后台
- 移动端原生 App

## 3. 目标用户

### 3.1 用户画像 A — 医疗饮食需求者

- **角色**：糖尿病患者、乳糜泻患者、肾病饮食限制者等
- **核心诉求**：精确排除含特定成分的餐厅，需要成分级别的信息（而非仅标签）
- **痛点**：现有平台缺乏成分详情，外出就餐存在健康风险；过滤后结果几乎为空

### 3.2 用户画像 B — 主动健康管理者

- **角色**：健身人群（增肌/减脂）、生酮/低碳饮食践行者、间歇性断食者
- **核心诉求**：按营养目标（高蛋白、低GI、低热量）筛选餐厅，了解菜品宏量营养素
- **痛点**：无法用自然语言描述营养目标进行搜索；现有平台无营养数据维度

### 3.3 用户画像 C — 价值观驱动的饮食选择者

- **角色**：素食主义者（vegan/vegetarian）、清真饮食者（halal）、有机饮食爱好者
- **核心诉求**：快速找到符合饮食信仰或价值观的餐厅，不浪费时间在无关结果上
- **痛点**：搜索"素食"返回大量仅有1-2道素菜的普通餐厅，无法区分纯素与弹性素

### 3.4 用户画像 D — 多人出行协调者

- **角色**：需要为饮食需求各异的同行者共同选餐厅的用户
- **核心诉求**：输入多套饮食限制，找到能同时满足所有人的餐厅
- **痛点**：需要逐一搜索再手动比对，没有工具支持"集体兼容"需求

## 5. 功能需求

### 5.1 数据层：餐厅与菜品数据集

#### 5.1.1 数据字段规范

**餐厅文档（Restaurant Document）：**

| 字段 | 类型 | 说明 |
|------|------|------|
| `restaurant_id` | string | 唯一标识 |
| `name` | string | 餐厅名称（多语言） |
| `cuisine_types` | string[] | 菜系标签（如 Italian, Sichuan, Mediterranean） |
| `diet_labels` | string[] | 饮食类型标签（见 5.1.2） |
| `allergen_free` | string[] | 明确声明不含的过敏原 |
| `description` | string | 餐厅介绍（全文检索主要来源） |
| `address` | string | 地址 |
| `geo` | {lat, lng} | 坐标（用于距离排序） |
| `price_level` | int (1-4) | 价格档次 |
| `rating` | float | 综合评分 |
| `hours` | object | 营业时间 |
| `menu_items` | MenuItem[] | 菜品列表（嵌套文档） |

**菜品文档（MenuItem）：**

| 字段 | 类型 | 说明 |
|------|------|------|
| `item_id` | string | 菜品 ID |
| `name` | string | 菜品名 |
| `description` | string | 菜品描述 |
| `diet_labels` | string[] | 该菜品的饮食标签 |
| `allergens` | string[] | 含有的过敏原 |
| `nutrition` | object | 营养成分（calories, protein, fat, carbs） |
| `price` | float | 单价 |

#### 5.1.2 饮食类型分类体系（Taxonomy）

系统预定义以下饮食类型，作为 Faceted 搜索的 Facet 维度：

**价值观/信仰类：**
- `vegan` — 纯素食（无任何动物产品）
- `vegetarian` — 素食（含蛋奶）
- `halal` — 清真
- `kosher` — 犹太洁食
- `organic` — 有机食材

**医疗/过敏类：**
- `gluten-free` — 无麸质
- `dairy-free` — 无乳制品
- `nut-free` — 无坚果
- `shellfish-free` — 无贝壳类海鲜
- `soy-free` — 无大豆

**健康目标类：**
- `low-carb` — 低碳水
- `keto` — 生酮（超低碳水+高脂）
- `high-protein` — 高蛋白
- `low-calorie` — 低热量（< 500 kcal 主菜为主）
- `low-sodium` — 低钠

#### 5.1.3 数据来源策略

- v1.0 使用静态数据集：从公开数据集（Yelp Open Dataset / OpenStreetMap + 人工标注）构建初始语料库
- 菜品级别数据通过半自动标注流程生成（初期 ≥ 60% 餐厅有菜单数据）
- 数据以 JSON 格式存储，索引构建时预处理

---

### 5.2 索引层：文档索引构建

#### 5.2.1 索引结构设计

系统维护两个独立索引：

**主索引（Restaurant Index）：**
- 索引粒度：餐厅文档
- 全文索引字段：`name`、`description`、`cuisine_types`（展开为文本）、`diet_labels`（展开为文本）、菜品名称集合（聚合）
- Facet 索引字段：`diet_labels`、`cuisine_types`、`price_level`、`rating`（分桶）
- 数值索引字段：`rating`、`price_level`、`geo`（用于地理距离计算）

**菜品子索引（MenuItem Index）：**
- 索引粒度：菜品文档（含所属餐厅 ID）
- 全文索引字段：`name`、`description`、`diet_labels`
- 用于"菜品级别搜索"模式

#### 5.2.2 文本预处理流水线

```
原始文本
  → 语言检测（英文 / 中文分支处理）
  → 分词（英文: whitespace + punctuation; 中文: jieba）
  → 小写归一化（英文）
  → 停用词去除（自定义停用词表，保留饮食关键词如 "no", "free", "without"）
  → 词干提取 / 词形还原（英文: NLTK PorterStemmer 或 spaCy Lemmatizer）
  → 同义词扩展（饮食同义词词典，见 5.3.2）
  → 建立倒排索引（term → posting list with TF）
```

#### 5.2.3 索引技术选型

- **索引引擎**：Elasticsearch 8.x（主选）或 Apache Solr
  - 原因：内置 BM25、支持 Facet、支持地理距离、支持高亮
- **备选**（轻量自建用于课程展示）：Whoosh（纯 Python）或 Tantivy
- 索引需支持增量更新（新增/修改餐厅不重建全量索引）

### 5.3 查询处理层

#### 5.3.1 查询解析流水线

```
用户输入文本
  → 语言检测
  → 分词 + 停用词过滤（与索引时保持一致）
  → 实体识别：提取地点、价格意图、饮食类型意图
  → 意图分类：
      - 饮食限制查询（"gluten free restaurants"）
      - 营养目标查询（"high protein low fat"）
      - 菜系查询（"Italian vegan pizza"）
      - 混合查询（以上组合）
  → 查询扩展（见 5.3.2）
  → 构建结构化查询（Boolean + BM25 组合）
```

#### 5.3.2 查询扩展策略

**饮食同义词词典（必须维护的核心资产）：**

| 规范术语 | 同义词/别名 |
|---------|------------|
| `vegan` | plant-based, no meat, no animal products, 纯素, 植物性饮食 |
| `vegetarian` | veggie, no meat, 素食, 斋菜 |
| `gluten-free` | gluten free, celiac safe, wheat-free, 无麸质, 无小麦 |
| `halal` | halal certified, 清真, 穆斯林友好 |
| `high-protein` | protein rich, high protein, 高蛋白, 增肌餐 |
| `low-carb` | low carbohydrate, carb free, 低碳, 低碳水 |
| `keto` | ketogenic, keto friendly, 生酮饮食 |
| `dairy-free` | lactose free, no milk, no dairy, 无乳糖, 无奶 |

**扩展策略：**
- 查询词命中词典时，自动 OR 扩展同义词（boost 原词 1.5x，同义词 1.0x）
- 拼写纠错：编辑距离 ≤ 2 时提示"您是否要搜索..."

#### 5.3.3 查询结构设计

标准查询为 Bool 查询组合：
```
{
  must: [全文匹配(query_text, fields=[name^3, description, diet_labels^2, menu_items.name^1.5])],
  filter: [Facet过滤(diet_labels), 价格范围过滤, 营业中过滤(可选)],
  should: [地理距离衰减(geo), 评分提升(rating^0.5)],
  minimum_should_match: 0
}
```

### 5.4 排序层：相关性评分

#### 5.4.1 基础排序模型

采用 **BM25** 作为基础文本相关性打分（参数 k1=1.2, b=0.75），在 Elasticsearch 中内置支持。

#### 5.4.2 多维度排序融合

最终分数 = 加权线性融合：

```
final_score = α × text_score(BM25)
            + β × diet_match_score     # 饮食标签精确匹配加分
            + γ × rating_score         # 评分归一化后的贡献
            + δ × distance_decay       # 距离衰减（高斯函数）
            + ε × personalization_boost  # 个性化因子（有档案时启用）
```

参数默认值（可通过 UI 切换排序模式调整权重）：

| 排序模式 | α | β | γ | δ | ε |
|---------|---|---|---|---|---|
| 综合相关性（默认）| 0.4 | 0.3 | 0.15 | 0.1 | 0.05 |
| 饮食匹配优先 | 0.2 | 0.6 | 0.1 | 0.05 | 0.05 |
| 评分优先 | 0.3 | 0.2 | 0.4 | 0.05 | 0.05 |
| 距离优先 | 0.3 | 0.2 | 0.1 | 0.35 | 0.05 |

#### 5.4.3 饮食匹配评分（diet_match_score）

```
diet_match_score = 
  Σ(exact_label_match × 1.0)       # 用户查询标签与餐厅标签完全匹配
  + Σ(menu_item_match × 0.6)       # 有菜品级别标签匹配（无餐厅级标签时）
  + allergen_safe_bonus × 0.5      # 餐厅声明不含用户标注的过敏原
  - allergen_risk_penalty × 2.0    # 含有用户标注的过敏原（严重惩罚）
```

#### 5.4.4 个性化重排序

- 用户设置饮食档案后，其档案标签注入查询的 `should` 子句，boost 个人偏好标签
- 用户的历史点击与"相关"反馈作为隐式正样本，更新个人排序权重（轻量 Learning to Rank）
- 个人数据仅影响当前用户，隔离存储

### 5.5 搜索结果展示

#### 5.5.1 搜索结果列表

每条结果展示：
- 餐厅名称 + 菜系标签
- 饮食标签徽章（色码区分类别）
- 相关性解释片段：高亮显示查询词命中位置（标题/描述/菜品名）
- 评分（星级）+ 价格等级 + 距离（有位置时）
- 饮食匹配度指示条（图形化显示与用户查询的匹配程度）

#### 5.5.2 Faceted 导航面板

左侧/顶部显示可交互的 Facet 面板：
- **饮食类型**：多选复选框，附带文档频次（如"vegan (47)"）
- **菜系**：多选，Top-10 显示，余下折叠
- **价格区间**：$ / $$ / $$$ / $$$$
- **评分**：4星以上 / 3星以上
- 选中的 Facet 以标签形式显示于搜索框下方，支持单独移除

#### 5.5.3 搜索结果解释（Explainability）

每条结果可展开"为什么推荐这家"面板：
- BM25 命中词列表及各词权重
- 饮食标签匹配情况（已匹配 / 未匹配 / 冲突）
- 如有过敏原风险，高亮警告提示

#### 5.5.4 搜索建议（Autocomplete）

- 用户输入 2 个字符后触发，实时展示建议
- 建议来源：饮食类型词典（精确匹配优先）+ 高频搜索词
- 有饮食档案时，优先展示档案匹配的建议

---

### 5.6 用户饮食档案

- 用户注册后可设置饮食档案（非必须，匿名用户也可搜索）
- 档案字段：饮食类型（多选）、过敏原（多选）、价格偏好、菜系偏好
- 档案影响：自动预填 Facet 过滤、调整排序权重、定制 Autocomplete
- 档案可随时修改或清空

---

### 5.7 搜索反馈与评估

- 用户可对结果标记"相关"或"不相关"（可选操作，非强制）
- 反馈数据用于计算系统级评估指标（Precision, Recall, NDCG）
- 管理界面展示检索评估报告（定期生成）

## 6. 非功能需求

### 6.1 性能

- 搜索响应时间：P99 ≤ 500ms（含查询扩展 + 检索 + 排序）
- Autocomplete 响应时间：P99 ≤ 100ms
- 索引构建（1,000 家餐厅）：≤ 10 分钟（离线批处理）
- 支持并发：≥ 50 QPS（v1.0 阶段）

### 6.2 可用性

- 系统可用性 SLA：≥ 99%（排除计划维护）
- 搜索无结果时，展示有意义的空结果页（推荐放宽筛选条件的建议）
- 拼写错误时，主动提示纠正，而非直接返回空结果

### 6.3 可扩展性

- 索引架构支持扩展至 100,000+ 餐厅文档，不需重写核心逻辑
- 饮食分类体系支持动态扩展新类别，不影响现有检索逻辑
- 排序权重参数可配置，不写死在代码中

### 6.4 准确性与公平性

- 过敏原信息展示必须标注数据来源与时效，提示"请向餐厅确认"
- 不因数据标注不完整而对餐厅产生系统性偏见（缺失字段不参与惩罚）
- 饮食类型标签严格按分类体系标准赋予，不模糊处理

### 6.5 安全与隐私

- 用户饮食档案（含过敏原信息）属敏感数据，加密存储
- 匿名用户的搜索日志不关联个人身份
- 不将用户饮食档案用于广告投放

## 7. 范围与边界

### 在范围内
- 餐厅与菜品文档的索引构建与全文检索
- 饮食类型 Faceted 搜索与过滤
- 查询解析、同义词扩展、拼写纠错

- 移动端原生 App
- 营养计划生成

## 8. 边界情况与开放问题

### 边界情况

| 场景 | 处理方式 |
|------|----------|
| 用户查询中含矛盾限制（如"vegan beef burger"）| 优先 vegan 标签，返回结果同时展示警告提示 |
| 餐厅未标注饮食标签 | 不展示饮食标签徽章，不参与饮食匹配评分，仅文本相关性排序 |
| 用户含严重过敏原的餐厅出现在结果中 | 前端显示红色警告徽章，不过滤（避免产生信息遮蔽），但加重惩罚项压低排名 |
| 搜索词为纯地名（无饮食关键词）| 降级为通用餐厅搜索，提示"未检测到饮食需求，已返回附近餐厅" |
| 多人模式取交集后无结果 | 展示"无法同时满足所有需求"提示，并建议最接近的近似结果（放宽某一限制后的结果） |

### 开放问题

1. **评估数据集**：人工标注的查询-相关文档对从哪里获取？是否使用专家标注或众包？
2. **地理位置**：v1.0 如何处理用户位置？是否需要浏览器定位权限？还是手动输入城市？
3. **多语言支持优先级**：v1.0 是否同时支持中英双语检索，还是先聚焦英文？
4. **菜品 vs 餐厅的检索粒度**：默认以餐厅为结果单元还是菜品为结果单元？
5. **营养数据来源**：菜品营养数据从哪里获取？USDA 数据库映射？商家自填？
6. **数据集版权**：使用 Yelp Open Dataset 时需遵守其学术使用协议，需确认。

## 9. 依赖关系

| 依赖项 | 用途 | 风险 |
|--------|------|------|
| Elasticsearch 8.x | 核心索引与检索引擎 | 需部署维护；授权与资源消耗需评估 |
| Yelp Open Dataset / 自建数据集 | 餐厅与菜品语料库 | 数据质量参差不齐，需人工清洗与标注 |
| spaCy / NLTK | 文本预处理（分词、词形还原） | 版本兼容性；中文需额外集成 jieba |
| jieba | 中文分词 | 饮食领域词语需添加自定义词典 |
| USDA FoodData Central API | 菜品营养数据补充 | API 限流；菜品名称映射准确性 |
| 浏览器 Geolocation API | 用户位置（距离排序） | 用户可能拒绝授权；需降级处理 |
| PostgreSQL | 用户档案、收藏夹、反馈数据存储 | 标准依赖，风险低 |
| Redis | Autocomplete 缓存、热门搜索词缓存 | 标准依赖，风险低 |
