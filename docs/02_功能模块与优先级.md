#### 2.3.1 LLM 辅助查询流程说明

当用户输入的食物名称在静态数据库中**未命中**时，触发 LLM 辅助路径：

```
用户输入食物名（未命中 DB）
    → 调用 LLM（优先 DeepSeek / Qwen，成本低）
    → Prompt：
        "请给出「{food_name}」每 100g 的营养成分估算：
         热量(kcal)、蛋白质(g)、脂肪(g)、碳水化合物(g)、
         适用饮食标签（从列表中选择：vegan/vegetarian/gluten-free/...）。
         仅返回 JSON，不要解释。"
    → 解析 LLM 返回 JSON
    → 展示给用户确认："找到了这条估算数据，确认后加入本地数据库"
    → 用户确认 → 写入扩展库（标记来源为 llm_inferred）
    → 后续同名食物直接从扩展库命中，不再调 LLM
```

**关键约束：**
- LLM 查询结果**必须经用户确认**才能入库（避免幻觉数据污染）
- 入库记录标注 `source: "llm_inferred"`，区别于静态库的 `source: "static"`
- 同一食物 LLM 查询只触发一次，结果缓存在本地，不重复调用

### 2.4 餐厅数据采集（Crawler Module）

| 子功能 | 优先级 | 状态 | 说明 |
|--------|--------|------|------|
| **高德地图 POI 爬虫** | P0 | 🔲 待实现 | 优先级最高；官方 API 有免费额度，法律风险最低；字段：餐厅名 / 分类 / 坐标 / 评分 / 地址 |
| **大众点评爬虫** | P1 | 🔲 待实现 | 含评论与菜品数据，是饮食标签的主要来源；反爬较强，需 User-Agent 轮换 + 请求限速 |
| **美团官网爬虫** | P1 | 🔲 待实现 | 菜品价格数据较全；需 Cookie 维持登录态 |
| 爬取字段标准化（映射到统一 Restaurant 文档结构） | P0 | 🔲 待实现 | 三个来源字段不同，需统一清洗 |
| 饮食标签自动推断（基于菜品名 / 描述 NLP） | P1 | 🔲 待实现 | 爬取的原始数据无饮食标签，需 NLP 后处理 |
| 去重合并（同一餐厅在多平台出现） | P1 | 🔲 待实现 | 按餐厅名 + 坐标近邻去重 |
| 增量爬取（仅更新新增 / 变更餐厅） | P2 | 🔲 待实现 | v1.0 先做全量离线爬取 |
| 爬取调度（定期刷新数据） | P2 | 🔲 待实现 | v1.0 手动触发 |

#### 2.4.1 爬虫优先级与策略说明

| 来源 | 接入方式 | 主要贡献数据 | 反爬风险 |
|------|----------|-------------|---------|
| 高德地图 | 官方 Places API（免费配额） | 坐标 / 分类 / 营业时间 / 基础评分 | 低（官方 API） |
| 大众点评 | HTML 爬虫（Playwright） | 菜品名称 / 用户评论 / 特色标签 | 高（JS 渲染 + 登录墙） |
| 美团 | HTML 爬虫（requests + 解析） | 菜单 / 价格 / 促销信息 | 中（部分接口需 Cookie） |

**v1.0 爬虫最小化策略：**
- 先用高德 API 建立餐厅骨架数据（坐标、名称、分类）
- 大众点评和美团爬虫作为数据增强，补充菜品与饮食标签
- 单次爬取间隔 ≥ 3 秒，设 User-Agent 池，规避 IP 封禁

## 3. 发布节奏规划

### v1.0 — MVP（当前目标）

**目标**：打通完整检索链路，用静态数据验证 IR 核心逻辑。

**必须完成（P0）：**
- 用户注册 / 登录 / 饮食档案
- 食物静态数据库 + 手动饮食录入 + 热量计算
- 高德 API 采集基础餐厅数据 → Elasticsearch 索引构建
- BM25 全文检索 + Faceted 饮食类型过滤
- 搜索结果页（含饮食标签徽章、过敏原警告）

**v1.0 交付时的数据规模目标：**
- 食物数据库：≥ 500 条常见食物（静态）
- 餐厅索引：≥ 500 家（高德 API 单城市爬取）

### v1.1 — 数据增强 & 搜索质量提升

**新增（P1）：**
- LLM 辅助食物热量查询（未收录食物触发）
- 大众点评 / 美团爬虫上线（补充菜品与饮食标签）
- 饮食标签 NLP 自动推断
- 多维度排序融合
- 查询扩展（同义词词典）+ 拼写纠错
- Autocomplete
- 个性化重排序（基于饮食档案）
- 饮食进度追踪

### v1.2 — 体验优化（规划中）

**新增（P2）：**
- 菜品子索引（菜品级别检索）
- 用户相关性反馈 + Learning to Rank
- 多人模式（多饮食需求取交集）
- 收藏餐厅
- 增量爬取调度

## 4. 模块依赖关系

```
用户体系
    └─► 所有模块（登录后才能保存档案、日志、收藏）

食物数据库（静态预置库）
    └─► 手动饮食记录（搜索食物的数据源）

LLM 辅助查询
    └─► 食物数据库（未命中时触发，查询结果经确认后扩充数据库）

手动饮食记录
    └─► 饮食目标追踪（累计摄入 vs 目标）

个人饮食档案
    ├─► IR 核心引擎（个性化排序的输入）
    └─► 前端过敏原警告（档案中的过敏原 → 结果中标红）

餐厅数据采集（Crawler）
    └─► IR 核心引擎（索引构建的数据来源）

高德 POI 爬虫（地理数据）
    └─► 大众点评 / 美团爬虫（以高德坐标为基准去重合并）

IR 核心引擎
    └─► 餐厅检索前端（检索 API 的后端实现）

用户相关性反馈
    └─► Learning to Rank（反馈数据驱动排序优化，v1.2）
```

## 5. 技术选型汇总

| 层次 | 选型 | 备注 |
|------|------|------|
| 索引与检索 | Elasticsearch 8.x | 内置 BM25、Facet、地理距离 |
| 文本预处理 | spaCy（英文）+ jieba（中文） | |
| 食物数据库（静态） | SQLite（开发）/ PostgreSQL（生产） | 轻量，无需独立服务 |
| LLM 辅助查询 | DeepSeek API / Qwen API | 优先低成本模型；仅用于食物热量估算 |
| 爬虫 | 高德官方 API + Playwright（大众点评）+ requests（美团） | |
| 用户数据 | PostgreSQL | 档案 / 日志 / 收藏 / 反馈 |
| 缓存 | Redis | Autocomplete 缓存 / LLM 查询结果缓存 |
| 前端框架 | React + TypeScript + Ant Design | 与 mingce-zhijian 前端技术栈保持一致 |
