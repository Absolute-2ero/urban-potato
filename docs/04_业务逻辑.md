## 1. 核心业务规则总览

| 规则编号 | 规则描述 | 涉及模块 |
|----------|----------|----------|
| BR-01 | 所有写操作（日志、档案、收藏、反馈）需登录鉴权，未登录返回 401 | 全局 |
| BR-02 | 用户只能读写自己的饮食档案、日志、收藏数据 | 饮食管理 |
| BR-03 | LLM 查询返回的食物数据必须经用户确认后方可入库（`verified=0` 不可用于计算） | 食物数据库 |
| BR-04 | 同一食物名（`name_zh` 精确匹配）在 SQLite 中只能存在一条记录，LLM 查询结果不重复入库 | 食物数据库 |
| BR-05 | 匿名用户可执行搜索和查看餐厅详情，不可保存日志、收藏、反馈 | 搜索、饮食管理 |
| BR-06 | 爬虫采集数据必须经过清洗与去重后才写入 Elasticsearch，不直接写入原始数据 | 爬虫 |
| BR-07 | 饮食日志中 `calories_kcal` 必须在写入时计算并存储（冗余），不依赖查询时计算 | 饮食管理 |
| BR-08 | 过敏原匹配为硬性逻辑：含有用户档案中过敏原的餐厅，前端必须显示警告，不论其排名 | 搜索结果 |
| BR-09 | `diet_labels` 的值只能来自预定义分类体系（见需求文档 5.1.2），拒绝任意字符串 | 档案、索引 |
| BR-10 | 搜索评分融合公式的权重参数从配置文件读取，不硬编码在代码中 | IR 引擎 |

## 2. 用户体系

### 2.1 注册流程

```
输入：username, password, email

1. 校验 username 唯一（查 users 表）
   └─ 已存在 → 返回 400 "用户名已被占用"

2. 校验 password 强度（≥8位，含字母+数字）
   └─ 不合规 → 返回 400 "密码需至少8位，包含字母和数字"

3. bcrypt 哈希 password（cost=12）

4. INSERT INTO users

5. INSERT INTO diet_profiles（空档案，user_id 关联）

6. 写入 Session，返回注册成功（用户已自动登录）
```

### 2.2 登录流程

```
输入：username, password

1. SELECT * FROM users WHERE username = ?
   └─ 不存在 → 返回 401

2. bcrypt 比对 password vs password_hash
   └─ 不匹配 → 返回 401（不区分"用户名错误"和"密码错误"）

3. 写入 Session（user_id）
4. 更新 users.last_login_at = now()
5. 返回 200
```

### 2.3 鉴权中间件

```python
# FastAPI Dependency
async def get_current_user(session: Session = Depends(get_session)):
    user_id = session.get("user_id")
    if not user_id:
        raise HTTPException(status_code=401)
    user = await db.get_user(user_id)
    if not user:
        raise HTTPException(status_code=401)
    return user
```

## 3. 饮食管理

### 3.1 食物搜索流程

```
输入：query（用户输入的食物名）

Step 1：SQLite FTS5 全文检索
  SELECT * FROM food_fts
  WHERE food_fts MATCH '{query}*'
  ORDER BY rank LIMIT 10

  ├─ 命中（results.count > 0）
  │   → 过滤 verified = 1（排除未确认的 LLM 数据）
  │   → 返回食物列表（含营养数据）
  │
  └─ 未命中（results.count = 0）
      → 触发 LLM 辅助查询（Step 2）

Step 2：LLM 辅助查询（BR-03）
  检查缓存：Redis GET food:llm:{query_normalized}
  ├─ 缓存命中 → 直接返回缓存的 LLM 结果（不再调用 LLM）
  └─ 缓存未命中：
      → 调用 LLM API（DeepSeek / Qwen）
      → Prompt 见业务逻辑 3.2
      → 解析 JSON 响应
      → Redis SET food:llm:{query_normalized} TTL=7天
      → 返回"估算数据"给前端，附带确认弹窗提示

Step 3：用户确认入库（BR-03）
  用户点击"确认并使用"：
  → 检查 food_items WHERE name_zh = ? 是否已存在（BR-04）
     ├─ 已存在 → 直接返回已有记录，不重复插入
     └─ 不存在 → INSERT food_items(source='llm_inferred', verified=0)
                → 用户确认后 UPDATE SET verified=1
                → 返回新 food_id
```

### 3.2 LLM 查询 Prompt 设计

```
System: 你是一个营养数据库助手，只能返回 JSON，不得包含解释文字。

User:
请给出「{food_name}」每 100g 的营养成分估算值。
返回格式（严格遵守）：
{
  "name_zh": "食物中文名",
  "name_en": "food name in English",
  "calories": 数字,
  "protein_g": 数字,
  "fat_g": 数字,
  "carb_g": 数字,
  "diet_labels": ["vegan"|"vegetarian"|"gluten-free"|"dairy-free"|"halal"|"keto"|"high-protein"|"low-carb"]
}
diet_labels 只从以上选项中选择适用的，不适用则返回空数组。
如果不确定该食物，calories 设为 -1。
```

**失败处理：**
- LLM 返回 JSON 解析失败 → 返回"无法获取该食物的营养数据，请手动输入"
- LLM 返回 `calories: -1` → 显示"该食物数据不确定，建议手动输入"
- LLM API 超时（>10s）→ 返回"查询超时，请稍后重试"

### 3.3 饮食日志录入流程

```
输入：food_id, quantity_g, meal_type, log_date（默认今日）

1. 鉴权（BR-01）

2. 从 SQLite 读取 food_items WHERE id = food_id AND verified = 1
   └─ 不存在或 verified=0 → 返回 400

3. 计算营养数据：
   calories_kcal = food.calories × quantity_g / 100
   protein_g     = food.protein_g × quantity_g / 100
   fat_g         = food.fat_g × quantity_g / 100
   carb_g        = food.carb_g × quantity_g / 100

4. INSERT INTO food_log_entries（BR-07：写入时计算，不依赖实时计算）

5. 返回新条目（含 id、计算后的营养数据）
```

### 3.4 每日饮食进度计算

```
输入：user_id, date

1. SELECT SUM(calories_kcal), SUM(protein_g), SUM(fat_g), SUM(carb_g)
   FROM food_log_entries
   WHERE user_id = ? AND log_date = ?

2. SELECT calorie_goal, protein_goal_g, fat_goal_g, carb_goal_g
   FROM diet_profiles
   WHERE user_id = ?

3. 计算各指标完成比例（completed / goal × 100%）
   └─ goal 为 NULL → 该指标不展示进度条，仅展示摄入量

4. 返回聚合结果
```

## 4. 餐厅数据采集（Crawler）

### 4.1 采集总流程

```
触发方式：手动触发（v1.0）/ 定时任务（v1.2）

Step 1：高德地图 POI 采集（P0，最先执行）
  → 调用高德 Place Search API（关键词="餐厅"，指定城市）
  → 返回 POI 列表（name, location, address, type, rating）
  → 标准化为 Restaurant 文档骨架
  → 写入临时存储（JSON 文件 / 内存）

Step 2：大众点评爬虫（P1，在高德骨架基础上补充）
  → 以高德 POI 餐厅名为关键词，在大众点评搜索
  → Playwright 渲染页面，提取：
      - 菜品列表（menu_items）
      - 用户标签（#素食 #适合聚餐 等）
      - 综合评分
  → 匹配到同名餐厅后合并字段

Step 3：美团爬虫（P1，补充菜品价格）
  → requests 请求美团搜索页（带 Cookie）
  → 提取菜品价格、促销信息
  → 合并到对应餐厅文档

Step 4：数据清洗 & 标准化
  → 去重（同名 + 坐标距离 < 50m 视为同一家）
  → NLP 饮食标签推断（见 4.2）
  → 字段完整性校验

Step 5：写入 Elasticsearch
  → bulk index（每批 100 条）
  → 记录爬取日志（来源、数量、时间）
```

### 4.2 饮食标签 NLP 推断规则

爬取的原始数据无饮食标签，通过以下规则自动推断：

```
规则匹配（优先级从高到低）：

1. 餐厅名称关键词匹配：
   "素食" | "蔬食" | "vegetarian" → diet_labels += "vegetarian"
   "纯素" | "vegan" → diet_labels += "vegan"
   "清真" | "halal" | "穆斯林" → diet_labels += "halal"
   "无麸质" | "gluten-free" → diet_labels += "gluten-free"

2. 用户标签匹配（大众点评抓取的 # 标签）：
   "#素食" → vegetarian；"#纯素" → vegan
   "#清真" → halal；"#健身" → high-protein

3. 菜品名称聚合分析：
   菜品中 70%+ 包含"豆腐"|"蔬菜"|"素" → vegetarian
   所有菜品均无"猪"|"牛"|"羊"|"鸡"|"鱼"|"虾" → vegan 候选
   （候选标签置信度较低，标注为 inferred，非 confirmed）

4. 置信度标注：
   diet_labels_source: "name_match" | "tag_match" | "menu_inferred"
   （前端可据此决定是否显示置信度说明）
```

### 4.3 爬虫限速与反爬规则（BR-06）

| 来源 | 请求间隔 | User-Agent 策略 | 异常处理 |
|------|---------|----------------|----------|
| 高德 API | 无限制（官方接口） | 固定 API Key | 429 时退避 60s |
| 大众点评 | ≥ 3 秒/请求 | 随机轮换 UA 池（10+条） | 403/429 时换 IP 或暂停 |
| 美团 | ≥ 2 秒/请求 | 随机 UA + Cookie 轮换 | 触发验证码时暂停并告警 |

## 5. 搜索业务逻辑

### 5.1 查询处理流程

```
输入：query_text, filters（diet_labels[], price_level[], rating_min）, geo（lat,lng）, sort_mode, page

Step 1：查询预处理
  → 语言检测（中文 / 英文 / 混合）
  → 分词（ik_smart 中文 / standard 英文）
  → 停用词过滤（保留否定词：无、不含、free、without）
  → 拼写纠错（编辑距离 ≤ 2，仅英文）

Step 2：查询扩展
  → 遍历分词结果，查饮食同义词词典
  → 匹配词 OR 扩展为同义词组（原词 boost=1.5，同义词 boost=1.0）

Step 3：构建 ES Bool Query
  must:   全文匹配（name^3, description, diet_labels_text^2, menu_items.name^1.5）
  filter: diet_labels（terms），price_level（terms），rating（range）
  should: geo_distance decay（high斯函数），rating boost

Step 4：ES 检索执行（BM25 排序）

Step 5：多维度重排序（Python 侧）
  final_score = α×text_score + β×diet_score + γ×rating_score + δ×distance_score
  （权重从配置文件读取，BR-10）

Step 6：过敏原标记（BR-08）
  对每条结果：检查 restaurant.allergens ∩ user_profile.allergens
  ├─ 有交集 → 在结果中附加 allergen_warning 字段（前端必须显示红色警告）
  └─ 无交集 → 正常展示

Step 7：格式化结果
  → 每条结果附带：highlights（高亮命中位置）、diet_match_detail、allergen_warning
  → 返回分页结果（默认 page_size=10）
```

### 5.2 饮食匹配评分计算

```python
def calc_diet_match_score(restaurant, query_diet_labels, user_allergens):
    score = 0.0

    # 正向：餐厅级标签命中
    for label in query_diet_labels:
        if label in restaurant.diet_labels:
            score += 1.0

    # 正向：菜品级标签命中（无餐厅级标签时的降级）
    if score == 0:
        menu_hit = any(
            label in item.diet_labels
            for item in restaurant.menu_items
            for label in query_diet_labels
        )
        if menu_hit:
            score += 0.6

    # 正向：声明不含用户过敏原
    for allergen in user_allergens:
        if allergen in restaurant.allergen_free:
            score += 0.5

    # 惩罚：含有用户过敏原（严重惩罚，BR-08）
    for allergen in user_allergens:
        if allergen in restaurant.allergens:
            score -= 2.0

    return max(score, -2.0)   # 最低 -2（含过敏原时）
```

## 6. 异常处理

### 6.1 食物数据库异常

| 异常 | 处理方式 |
|------|----------|
| LLM API 超时（>10s） | 返回 "查询超时"，不写缓存，用户可重试 |
| LLM 返回非 JSON | 记录错误日志，返回"无法解析营养数据" |
| LLM 返回 `calories: -1` | 显示"数据不确定"提示，不入库 |
| SQLite 文件损坏 | 启动时检测，损坏则从 seed 文件重建静态库 |

### 6.2 搜索异常

| 异常 | 处理方式 |
|------|----------|
| Elasticsearch 不可用 | 返回 503，前端展示"搜索服务暂时不可用" |
| 查询结果为空 | 返回 200 空列表 + 建议语（放宽筛选条件） |
| 排序计算异常 | 降级为 ES 原始 BM25 分数排序，记录告警日志 |
| 地理位置未提供 | 距离维度权重设为 0，不影响其他维度 |

### 6.3 爬虫异常

| 异常 | 处理方式 |
|------|----------|
| 大众点评返回 403 | 暂停该来源 30 分钟，继续处理其他来源 |
| 美团触发滑块验证码 | 暂停，记录告警，等待人工处理 |
| 高德 API 超出配额 | 停止当日采集，次日继续（配额按天重置） |
| 餐厅数据字段缺失 | 允许部分字段为空，不因缺失字段拒绝入库 |
